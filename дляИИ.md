# Шаблон: инструкция по созданию страниц раздела

Этот файл — универсальный шаблон для разработчиков и ИИ-ассистентов. Используйте его при создании или правке страниц в любом разделе. Цель: единообразие, доступность и безопасность контента.

---

## Краткие правила

- Имя файла: kebab-case, начинается с имени темы, например `async-await.html`, `functions-basics.html`.
- Формат страницы: HTML с partial includes или MD, в зависимости от проекта; структура должна быть семантической и иметь якорную навигацию.
- Код: только в блоке:
```html
<div class="code-block">
  <pre><code>...</code></pre>
</div>
```
- Запрещено: `grid-column-*`, `example`, любые нестандартные обёртки вокруг `<pre><code>`.
- Заголовки: ровно один H1 на страницу.
- Meta description: уникален, 120–160 символов.
- Доступность: `skip-link`, `nav role="navigation" aria-label="..."`, активная ссылка — `aria-current="page"`, `main tabindex="-1"`.

---

## Обязательная структура страницы

### Head / meta через partial
```html
@@include('../partials/head.html', {
  "title": "Заголовок — краткое пояснение",
  "description": "Meta description 120–160 символов"
})
```

### Skip link + header
```html
<a class="skip-link" href="#main">Перейти к содержимому</a>
@@include('../partials/header.html', {})
```

### Main + навигация по секциям
```html
<main class="page" id="main" tabindex="-1">
  <nav class="nav-tabs" id="page-nav" role="navigation" aria-label="Навигация по разделам страницы">
    <a href="#intro" class="active" aria-current="page">Введение</a>
    <a href="#concepts">Основные понятия</a>
    ...
  </nav>

  <!-- секции -->
</main>
```

### Footer / scripts
```html
@@include('../partials/footer.html', {})
@@include('../partials/popup.html', {})
@@include('../partials/js.html', {})
```

## Новое обязательное требование: определение в начале каждой секции

- Каждая логическая секция или подпункт обязаны начинаться с короткого определения темы в формате:
```html
<p class="note"><strong>Тема секции</strong> — краткое точное определение/цель секции.<br />Короткое дополнительное пояснение при необходимости.</p>
```
- Определение должно стоять перед любым списком, таблицей, кодом или блоком примера.
- Длина определения: 1–2 предложения.
- Это требование улучшает сканируемость, доступность и SEO.

---

## Рекомендуемые секции

1. Введение — 1–3 абзаца. Начинается с определения.
2. Основные понятия — краткие определения ключевых терминов. Каждый термин — отдельное определение.
3. Основной контент — логические подразделы (`h2/h3`). Каждый подраздел начинается с определения.
4. Примеры — короткие, 5–15 строк, с комментариями, каждый пример сопровождается определением.
5. Практика / интерактив — безопасные демо без `eval`. Начинать с определения цели практики.
6. Ловушки / типичные ошибки (`div.warning`) — каждую проблему вводить определением.
7. Советы / шаблоны (`div.note-block`) — каждый шаблон с определением.
8. Лучшие практики — каждый пункт с определением и примером.
9. Сводка / вывод — краткое определение итоговой мысли.

## Микро-структура подраздела

Для каждого понятия/метода/типа:

1. Определение — обязательно.
2. Особенности — краткий список ключевых свойств.
3. Примеры — код в `div.code-block`.
4. Применение — реальные сценарии использования.
5. Ловушки / рекомендации — если нужно.

Пример:
```html
<p class="note"><strong>Замыкания</strong> — функция вместе с лексическим окружением, в котором она была создана.<br />Используются для инкапсуляции состояния и создания приватных данных.</p>

<ul class="list-ul">
  <li>Особенности: сохраняют состояние, могут приводить к утечкам при неправильном использовании.</li>
</ul>

<div class="code-block">
  <pre><code>// Пример: простое замыкание
function makeCounter() {
  let count = 0;
  return function() { return ++count; };
}
const c = makeCounter();</code></pre>
</div>

<p>Применение: счётчики, фабрики функций, приватные переменные в модулях.</p>
```

## Стандартные классы и семантика

### Layout
- `main.page` — главный контейнер (`id="main"`, `tabindex="-1"`).
- `section.section` — логическая секция.
- `div.container` — внутренний отступ / ограничение ширины.

### Навигация
- `nav.nav-tabs#page-nav[role="navigation"]` — якорная навигация по странице.
- В ссылках: `class="active"` + `aria-current="page"` для текущего пункта.

### Тексты
- `p.note` — определения / короткие параграфы.
- `p.note.note--info` — советы / подсказки.
- `div.note-block` — крупные информационные блоки.
- `div.warning` — предупреждения / ловушки.
- `ul.list-ul` — списки.
- `table.table-note` — таблицы / сравнения.

### Изображения
- `figure.image-block` — общий контейнер для изображения и подписи.
- `img.image-block__img` — само изображение.
- `figcaption.image-block__caption` — подпись под изображением.
- Дополнительные модификаторы допускаются только если они уже есть в проекте, например `image-block--wide`, `image-block--small`, `image-block--center`.

### Правила для изображений
- Используйте `alt` всегда.
- Если изображение декоративное, `alt=""`.
- Если изображение несёт смысл, `alt` должен кратко и точно описывать содержимое.
- Не вставляйте изображение без контейнера, если в проекте принят единый стиль через `figure`.
- Если картинка должна быть на всю ширину контента, используйте соответствующий модификатор класса, а не inline-стили.
- Не дублируйте текст статьи в `alt`.
- Подпись в `figcaption` должна быть короткой и не повторять дословно основной абзац.
- Не ставьте важную информацию только в картинку без текстовой альтернативы.

### Код
- `div.code-block` вокруг `<pre><code>`.

### Интерактив
- `<button class="practice-btn" data-code="key">Название</button>`.
- Результат: `<div class="code-block"><pre><code id="output-result" aria-live="polite">// result</code></pre></div>`.

---

## Правила для кода и примеров

- Экранируйте спецсимволы: `<` → `&lt;`, `>` → `&gt;`, `&` → `&amp;`.
- Не используйте `grid-column-*`, `example` и другие кастомные обёртки.
- Комментарии в коде допустимы и приветствуются.
- Для демонстрации правильного и неправильного подхода используйте парные примеры с комментариями `// Правильно:` и `// Неправильно:`.
- Не запускайте код через `eval`. Для онлайн-демо используйте безопасные API или песочницы.

## Интерактивы и JS-паттерны

Рекомендуемый паттерн для кнопок-примеров:
```js
const examples = {
  key1: () => 'result',
  asyncKey: async () => {
    return 'ok';
  }
};

document.addEventListener('click', async (e) => {
  const btn = e.target.closest('[data-code], .practice-btn');
  if (!btn) return;

  const key = btn.getAttribute('data-code') || btn.dataset.code;
  const fn = examples[key];
  if (!fn) return;

  try {
    const res = fn();
    const output = document.getElementById('output-result');
    if (res instanceof Promise) {
      output.textContent = 'Выполняется...';
      output.textContent = await res;
    } else {
      output.textContent = res;
    }
  } catch (err) {
    document.getElementById('output-result').textContent = 'Ошибка: ' + (err && err.message ? err.message : String(err));
  }
});
```

- Все результаты интерактивов должны выводиться в элемент с `aria-live="polite"`.

## Accessibility

- Skip link: `<a class="skip-link" href="#main">Перейти к содержимому</a>`.
- `nav role="navigation" aria-label="Навигация по разделам страницы"`.
- На активной ссылке: `aria-current="page"`.
- `main` — `tabindex="-1"` для фокуса.
- Динамический вывод — `aria-live="polite"`.
- Используйте `<button>` для интерактивов, а не `<a href="#">`.

## SEO / meta

- Title и description через `head.html` partial.
- Description: 120–160 символов, уникален.
- Один H1 на страницу.
- По желанию: `rel="canonical"`, Open Graph и Twitter meta через partial.

---

## Примеры «правильно / неправильно»

Правильно:
```html
<div class="code-block">
  <pre><code>fetch('/api/data').then(r => r.json()).then(data => console.log(data));</code></pre>
</div>
```

Неправильно:
```html
<div class="example">
  <div class="grid-column-50">
    <pre><code>...</code></pre>
  </div>
</div>
```

→ Заменить на `div.code-block`.

## Чек-лист перед публикацией

- [ ] title + meta description уникальны.
- [ ] H1 только один и отражает тему.
- [ ] Каждая секция и каждый подпункт начинаются с определения в формате `<p class="note"><strong>Тема</strong> — ...</p>`.
- [ ] Все `<pre><code>` находятся в `div.code-block`.
- [ ] Спецсимволы экранированы.
- [ ] Нет `grid-column-*`, `example` или других нестандартных обёрток.
- [ ] Навигация содержит все секции и `id` совпадают.
- [ ] Интерактивы протестированы.
- [ ] Доступность: skip-link, `role="navigation"`, `aria-current`, `aria-live`.
- [ ] При переименовании настроен 301 редирект (`_redirects`) или правило Nginx.

## Процесс публикации

1. Подготовить черновик по шаблону и проверить локально.
2. Протестировать HTML, JS и доступность.
3. Обновить partials/subnav, если добавляете или переименовываете страницу.
4. Создать PR с описанием изменений; при переименовании добавить запись в `_redirects` или правило в конфигурацию сервера.
5. После ревью — мердж и проверка на проде.
