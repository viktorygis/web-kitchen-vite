Введение в API

API (Application Programming Interface - интерфейс программирования приложений) - это набор правил и протоколов, которые позволяют разным программам общаться друг с другом, это способ, с помощью которого твой проект общается с сервером, получает и отправляет данные.

Когда ты открываешь любой современный сайт: профиль пользователя, интернет-магазин или чат, эти данные чаще всего приходят с сервера по API.

Где это используется?

  • 📰 Новостные ленты, профили, чаты, списки товаров
  • 🗂️ Получение и отправка данных на сервер
  • 🌍 Интеграция с картами, погодой, другими внешними сервисами

Представь браузер как ресторан — ты (клиент) делаешь заказ (запрос), оффициант (API) передаёт его на кухню (сервер), откуда тебе возвращают результат (ответ)!

💡 Зачем нужны API?

Api нужно для - обмена данными между клиентом (твоим приложением) и сервером (где хранятся данные).
Это позволяет твоему приложению быть динамичным и интерактивным.

Что можно делать с API?

  • 📥 Загрузка данных: получение списка пользователей, товаров, сообщений
  • 📤 Отправка данных: регистрация, оформление заказа, публикация комментария
  • 🔄 Обновление данных: изменение профиля, статуса заказа
  • ❌ Удаление данных: удаление аккаунта, товара из корзины

Без API твой проект был бы статичным, без возможности взаимодействия с пользователем и сервером.

Альтернатива API?

Раньше для обмена данными использовались технологии вроде SOAP или XML-RPC, но сейчас они практически не используются из-за своей сложности и громоздкости. Современные веб-приложения в основном используют REST API с JSON.

📜 Как устроены современные API (REST)

Современные API - это чаще всего RESTful API, которые используют стандартные HTTP-методы и формат JSON для обмена данными.
Это делает их простыми в использовании и интеграции с фронтенд-приложениями, такими как Vue.

Современные API виды:

  • REST — самый популярный стиль API, использующий HTTP-методы и URL-адреса для взаимодействия с ресурсами
  • GraphQL (GraphQL - это способ запроса данных) — стиль API, позволяющий клиенту запрашивать только нужные данные
  • gRPC (gRPC - это способ обмена данными) — стиль API, использующий протоколы для высокопроизводительного обмена данными между сервисами

REST (Representational State Transfer - представление состояния передачи) — это архитектурный стиль для создания веб-сервисов, который делает взаимодействие между клиентом и сервером простым и понятным.

Принципы REST:

  1. Клиент-серверная архитектура — сервер хранит данные, фронтенд их запрашивает
  2. HTTP-методы — используются GET, POST, PUT, DELETE (чтоб читать и менять данные)
  3. JSON — стандарт передачи, простой текстовый формат
  4. Уникальные URL-адреса (endpoints) — обращаемся к нужным сущностям по конкретному адресу

🔎 Пример простого GET-запроса к API

Как получить список пользователей с сервера?

Чтобы получить данные, мы отправляем HTTP-запрос к определённому URL-адресу (endpoint):

GET https://api.example.com/users
Что здесь происходит?
  • GET — это HTTP-метод, означающий “получить данные”
  • https://api.example.com/users — это URL-адрес API
  • “users” — ресурс (список пользователей)

Ответ от сервера:

Сервер возвращает данные в формате JSON:

[
  { "id": 1, "name": "Алиса" },
  { "id": 2, "name": "Боб" }
]

Здесь мы получили массив объектов, каждый из которых представляет пользователя с уникальным id и name.

Итого: мы отправили GET-запрос к API и получили данные в формате JSON!

🔄 Основные HTTP-методы в REST

HTTP-методы — это команды, которые мы отправляем серверу, чтобы указать, что именно нужно сделать с данными.

Основные методы:

  • GET — получить данные с сервера
  • POST — отправить новые данные (например, создать пользователя или оформление заказа)
  • PUT — обновить существующие данные (например, изменить профиль)
  • DELETE — удалить данные

Важно: только GET “безопасен” и не меняет данные на сервере, остальные — изменяют!

🌐 Структура URL

Структура URL — это способ организации адреса, по которому можно получить данные с сервера.

Части URL:

  • Протокол — указывает, каким способом будет происходить обмен данными (например, `https://`)
  • Домен — имя сервера (например, `api.example.com`)
  • Путь — указывает на конкретный ресурс или коллекцию ресурсов (например, `/users` или `/users/1`)
  • Параметры запроса — дополнительные данные, передаваемые в URL для фильтрации или сортировки (например, `?age=25`)
  • Хэш-параметры — дополнительные данные, передаваемые после символа # в URL (например, `#section1`)
  • Хедеры — дополнительная информация, передаваемая в заголовках HTTP-запроса (например, `Authorization: Bearer token`)

Виды URL-адреса:

  • Уникальные URL-адреса — адреса, которые указывают на конкретный ресурс (например, `/users/1`)
  • Не уникальные URL-адреса — адреса, которые указывают на коллекцию ресурсов (например, `/users`)

🌐 Уникальные URL

Уникальные URL-адреса (endpoints - "конечная точка") — это уникальные адреса, по которым мы обращаемся к конкретным ресурсам на сервере.
Они нужны для того, чтобы точно указать, какие данные мы хотим получить или изменить.

Пример структуры уникального URL:

https://api.example.com/users/1
  • https://api.example.com — базовый URL сервера
  • /users — ресурс (список пользователей)
  • /1 — уникальный идентификатор конкретного пользователя

Таким образом, чтобы получить данные конкретного пользователя с ID 1, мы отправляем GET-запрос по этому URL.

🌐 Не уникальные URL

Не уникальные URL-адреса — это адреса, которые указывают на коллекцию ресурсов, а не на конкретный элемент.
Они используются для получения списка всех элементов определённого типа.

Пример структуры не уникального URL:

https://api.example.com/users
  • https://api.example.com — базовый URL сервера
  • /users — р��сурс (список пользователей)

Чтобы получить список всех пользователей, мы отправляем GET-запрос по этому URL.

🌐 Параметры запроса

Параметры запроса — это дополнительные данные, которые можно передать в URL для фильтрации или сортировки данных. Например: 

https://api.example.com/users?age=25&sort=name
  • ?age=25 — параметр фильтрации по возрасту
  • &sort=name — параметр сортировки по имени

Таким образом, мы можем получить только тех пользователей, которые соответствуют заданным критериям.

🌐 Хэш-параметры

Хэш-параметры — это дополнительные данные, которые передаются после символа # в URL. Они часто используются для навигации внутри страницы или для хранения состояния приложения. Например: 

https://api.example.com/users#section1
  • #section1 — хэш-параметр, указывающий на определённый раздел страницы
  • # — символ, обозначающий начало хэш-параметров
  • section1 — идентификатор раздела, берется из id HTML-элемента

Хэш-параметры не отправляются на сервер, они обрабатываются на стороне клиента (в браузере).

🌐 Хедеры

Хедеры — это дополнительная информация, которая передаётся в заголовках HTTP-запроса. Они могут содержать данные об авторизации, типе контента и другие метаданные. Например: 

Authorization: Bearer token
Content-Type: application/json
  • Authorization: Bearer token — хедер для авторизации с использованием токена
  • Content-Type: application/json — хедер, указывающий, что данные отправляются в формате JSON

Хедеры важны для правильной обработки запросов и ответов между клиентом и сервером.
