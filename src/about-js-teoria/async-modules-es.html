Модули в JavaScript (ES Modules)

Модули — это файлы с кодом, которые можно подключать к другим файлам через export и import.
Они позволяют разбивать программу на независимые части, которые легко переиспользовать, поддерживать и тестировать.

Зачем нужны модули:

  • Организация кода — проект делится на логические части
  • Повторное использование — один модуль подключается в разных местах
  • Изоляция — переменные модуля не попадают в глобальную область видимости
  • Упрощение тестирования — модуль можно тестировать независимо
  • Совместная работа — модули легко передавать между участниками команды

Основные понятия

  • Модуль — файл с кодом (функции, классы, переменные), который можно использовать в других файлах.
    По умолчанию модуль выполняется в строгом режиме ('use strict' не нужен).

  • export — делает функции, переменные или классы доступными за пределами файла.
    Без export всё содержимое модуля остаётся приватным.

  • import — позволяет использовать экспортированные значения из другого файла.
    Имена при именованном импорте должны совпадать с именами при экспорте.

  • type="module" — атрибут тега <script>, который сообщает браузеру, что это ES-модуль.
    Такие скрипты подключаются с поддержкой import/export и выполняются асинхронно (как с defer).

Экспорт (export)

Именованный экспорт — способ сделать доступными за пределами модуля сразу несколько значений, обращаясь к каждому по своему имени.

Особенности:

  • Можно экспортировать сколько угодно значений из одного файла
  • Экспортировать можно прямо при объявлении или одним блоком в конце файла
  • При импорте имена нужно указывать в фигурных скобках и они должны совпадать с экспортированными
  • Можно переименовать значение при экспорте через as

Примеры:

// math.js — экспорт прямо при объявлении
export const PI = 3.14;

export function square(x) {
  return x * x;
}
// math.js — экспорт одним блоком в конце файла
const PI = 3.14;

function square(x) {
  return x * x;
}

export { PI, square };
// math.js — переименование при экспорте
const PI = 3.14;

export { PI as Pi };

Применение:

  • Библиотеки утилит (математика, форматирование, валидация)
  • Константы конфигурации (URL API, ключи, лимиты)
  • Переиспользуемые компоненты и хуки

⚠ Без export переменные и функции остаются доступны только внутри своего файла.

Импорт (import)

import — позволяет подключить и использовать значения, экспортированные из другого модуля.

Особенности:

  • Путь к модулю указывается явно, обычно с расширением .js
  • Именованный импорт берётся в фигурные скобки { }
  • Можно импортировать всё содержимое модуля как объект через * as
  • Импортированные значения доступны только для чтения

Примеры:

// main.js — именованный импорт
import { add, PI } from './math.js';

console.log(add(2, 3)); // 5
console.log(PI); // 3.14
// main.js — импорт всего модуля как объекта
import * as math from './math.js';

console.log(math.add(2, 3)); // 5
console.log(math.PI); // 3.14

Применение:

  • Подключение сторонних библиотек и собственных утилит
  • Сборка приложения из независимых модулей-компонентов
  • Динамическая загрузка кода по требованию (через import())

Экспорт по умолчанию (default export)

Экспорт по умолчанию — способ экспортировать одно главное значение модуля без фигурных скобок при импорте.

Особенности:

  • В одном модуле может быть только один export default
  • При импорте имя можно указать любое — оно не обязано совпадать с оригинальным
  • Можно совмещать с именованными экспортами в одном файле

Примеры:

// user.js
export default class User {
  constructor(name) {
    this.name = name;
  }
}
// main.js — имя при импорте можно выбрать своё
import Person from './user.js';

const u = new Person("Аня");
console.log(u.name); // "Аня"

Применение:

  • Модуль с одним главным экспортируемым классом или компонентом
  • Точка входа модуля (например, основной React-компонент файла)

Подключение модулей в HTML

Чтобы использовать модули в браузере, скрипт нужно подключить через тег <script> с атрибутом type="module".

Особенности:

  • Модули выполняются в строгом режиме автоматически
  • Модули по умолчанию откладываются (ведут себя как скрипты с defer)
  • Модули работают только через HTTP(S) — открытие файла напрямую (file://) вызовет ошибку CORS

Примеры:

<script type="module" src="main.js"></script>

Применение:

  • Подключение точки входа приложения, собранного из ES-модулей
  • Постепенный переход со старых глобальных скриптов на модули

🔹 Атрибут type="module" — единственное, что отличает подключение модуля от обычного скрипта.

Примеры

Пример 1: математический модуль

// math.js
export function add(a, b) {
  return a + b;
}

export const PI = 3.14;
// main.js
import { add, PI } from './math.js';

console.log(add(2, 3)); // 5
console.log(PI); // 3.14

Пример 2: функция приветствия

// greet.js
export function greet(name) {
  return `Привет, ${name}!`;
}
// main.js
import { greet } from './greet.js';

console.log(greet("Вася")); // Привет, Вася!

Пример 3: импорт всего модуля

// utils.js
export function multiply(a, b) {
  return a * b;
}

export function subtract(a, b) {
  return a - b;
}

export const VERSION = "1.0";
// main.js
import * as utils from './utils.js';

console.log(utils.multiply(3, 5)); // 15
console.log(utils.subtract(10, 4)); // 6
console.log(utils.VERSION); // 1.0

Ловушки и типичные ошибки

Важные замечания:

  • Открытие HTML-файла с модулями напрямую через file:// вызывает ошибку CORS — нужен локальный сервер (например, Live Server, Vite).
  • Забытое расширение .js в пути импорта (import { add } from './math') приводит к ошибке — браузер, в отличие от сборщиков, не достраивает расширение сам.
  • В одном модуле может быть только один export default — повторное объявление вызовет синтаксическую ошибку.
  • Смешение CommonJS (require/module.exports) и ES-модулей (import/export) в одном файле невозможно без сборщика или настройки окружения.
  • Имена при именованном импорте должны точно совпадать с именами экспорта (с учётом регистра), иначе значение будет undefined.

Лучшие практики

Используй именованный экспорт для утилит с несколькими функциями и export default — для модуля с одним главным значением (классом, компонентом).

Не злоупотребляй import * as без необходимости — явный именованный импорт читается понятнее и упрощает поиск неиспользуемого кода.

Держи модули небольшими и сфокусированными на одной задаче — это упрощает тестирование и переиспользование.
