PATCH и DELETE запросы в Express.js

PATCH и DELETE запросы — это методы для частичного изменения и удаления данных на сервере. PATCH используется для частичного обновления ресурсов (обновляются только переданные поля), DELETE — для полного удаления ресурсов.
В отличие от PUT, который заменяет весь ресурс, PATCH позволяет обновлять только определенные свойства, сохраняя остальные без изменений.

В этом разделе вы научитесь:

  • Использовать PATCH запросы для частичного обновления данных.
  • Создавать обработчики DELETE для удаления ресурсов.
  • Различать PATCH и PUT для правильного использования.
  • Обрабатывать ошибки при частичном обновлении и удалении.
  • Реализовать полный REST API с поддержкой всех методов.

PATCH запросы

PATCH запрос — это HTTP метод для частичного обновления существующего ресурса. В отличие от PUT, который требует передачи всех данных, PATCH позволяет обновлять только те поля, которые были изменены. Остальные поля сохраняют свои текущие значения.

Характеристики PATCH:

  • Частичное обновление — обновляет только переданные поля, остальные остаются без изменений.
  • Данные в теле — информация передается в теле (body) запроса.
  • Требует ID — ID ресурса передается в URL пути (/resource/:id).
  • Статус ответа — обычно возвращает 200 (OK).
  • Идемпотентность зависит от реализации — часто может быть идемпотентным, но не всегда.

Пример PATCH запроса:

Здесь, если передан только параметр price, то только цена будет обновлена, а имя останется прежним.

app.patch('/update-product/:id', (req, res) => {
  const productId = req.params.id; // Получаем ID продукта из URL
  const updatedData = req.body; // Получаем обновленные данные из тела запроса

  // Это данные, которые были переданы в PATCH-запросе
  const product = {
    name: 'Телевизор',  // Имя остаётся без изменений
    price: updatedData.price || 5000, // Цена будет обновлена, если она передана
  };

  res.send(`Продукт с ID ${productId} был обновлён! Новые данные: ${JSON.stringify(product)}`);
});
Что происходит на сервере?
  1. Сервер получает ID продукта из URL, например, /update-product/1
  2. Затем он получает обновленные данные из тела запроса. Например, запрос может выглядеть так:
    3. { "price": 4500 }
  3. Сервер обновляет цену продукта на 4500, оставляя остальные данные без изменений.

Базовый пример PATCH запроса:

// Хранилище пользователей
let users = [
  { id: 1, name: 'Иван', email: 'ivan@mail.com', age: 25, phone: '+7-999-123-45-67' },
  { id: 2, name: 'Мария', email: 'maria@mail.com', age: 30, phone: '+7-999-987-65-43' }
];

// PATCH обработчик — частичное обновление пользователя
app.patch('/users/:id', (req, res) => {
  const userId = parseInt(req.params.id);
  const { name, email, age, phone } = req.body;

  // Найти пользователя по ID
  const user = users.find(u => u.id === userId);

  if (!user) {
    return res.status(404).json({
      error: 'Пользователь не найден'
    });
  }

  // Обновляем только переданные поля
  if (name !== undefined) user.name = name;
  if (email !== undefined) user.email = email;
  if (age !== undefined) user.age = age;
  if (phone !== undefined) user.phone = phone;

  user.updatedAt = new Date();

  res.json({
    message: 'Пользователь частично обновлен',
    user
  });
});

app.listen(3000);

Как отправить PATCH запрос:

Через fetch:
fetch('http://localhost:3000/users/1', {
  method: 'PATCH',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    age: 26
    // Обновляем только возраст, остальные поля остаются без изменений
  })
})
.then(res => res.json())
.then(data => console.log(data));
Через curl:
curl -X PATCH http://localhost:3000/users/1 \
  -H "Content-Type: application/json" \
  -d '{"age":26}'

Ответ на успешный PATCH запрос:

{
  "message": "Пользователь частично обновлен",
  "user": {
    "id": 1,
    "name": "Иван",
    "email": "ivan@mail.com",
    "age": 26,
    "phone": "+7-999-123-45-67",
    "updatedAt": "2024-07-14T11:45:00.000Z"
  }
}

Преимущество PATCH: Если обновляем только одно поле (например, только возраст), отправляем только это поле. Это экономит трафик и делает запрос более эффективным.

DELETE запросы

DELETE запрос — это HTTP метод для удаления существующего ресурса на сервере. Это необратимая операция, поэтому перед удалением рекомендуется подтвердить действие и проверить права доступа.

Характеристики DELETE:

  • Полное удаление — удаляет весь ресурс из системы.
  • Идентификатор в URL — ID ресурса передается в URL пути (/resource/:id).
  • Обычно без тела — DELETE запросы часто не содержат тело, только заголовки.
  • Идемпотентный — несколько одинаковых DELETE запросов приводят к одному результату (ошибка 404).
  • Статус ответа — обычно возвращает 204 (No Content) или 200 (OK).

Базовый пример DELETE запроса:

Предположим, у нас есть сервер, который управляет данными о товарах. Каждый товар имеет уникальный ID, и нам нужно удалить товар с конкретным ID. Для этого мы отправляем запрос: DELETE /delete-product/123 Здесь 123 — это ID товара, который мы хотим удалить.

// Подключаем express и создаём приложение
const express = require('express');
const app = express();

// Данные для примера, список продуктов с ID
let products = [
  { id: 1, name: 'Телефон', price: 500 },
  { id: 2, name: 'Ноутбук', price: 1000 },
  { id: 3, name: 'Камера', price: 300 },
];

// Обработка DELETE-запроса для удаления товара по ID
app.delete('/delete-product/:id', (req, res)= > {
  const productId = req.params.id; // Извлекаем ID из параметра URL
  const productIndex = products.findIndex(product= > product.id == productId); // Находим индекс товара в массиве

  // Проверяем, существует ли товар с таким ID
  if (productIndex === -1) {
    // Если товар не найден, отправляем ошибку 404
    return res.status(404).send('Товар с таким ID не найден!');
  }

  // Если товар найден, удаляем его из массива
  products.splice(productIndex, 1);

  // Отправляем сообщение об успешном удалении
  res.send(`Продукт с ID ${productId} был удалён!`);
});

// Запуск сервера на порту 3000
app.listen(3000, ()= > {
  console.log('Сервер запущен на порту 3000');
});
Что происходит на сервере
  1. Когда клиент отправляет DELETE-запрос с ID товара, сервер принимает запрос и извлекает ID через req.params.id
  2. Сервер ищет товар с этим ID в базе данных (в нашем случае — в массиве products).
  3. Если товар найден, он удаляется с помощью метода splice()
  4. Сервер отправляет клиенту ответ с сообщением, что товар был удалён. Если товар с таким ID не найден, сервер вернёт ошибку 404.
Что происходит на другом конце (на клиенте)
  1. Что происходит на клиенте
  2. Клиент отправляет запрос DELETE с ID товара, например, /delete-product/123
  3. Если товар с таким ID найден, клиент получит сообщение: "Продукт с ID 123 был удалён!"
  4. Если товар не найден, клиент получит ошибку 404 с сообщением "Товар с таким ID не найден!"

Базовый пример DELETE запроса:

// Хранилище пользователей
let users = [
  { id: 1, name: 'Иван', email: 'ivan@mail.com' },
  { id: 2, name: 'Мария', email: 'maria@mail.com' },
  { id: 3, name: 'Петр', email: 'petr@mail.com' }
];

// DELETE обработчик — удаление пользователя
app.delete('/users/:id', (req, res) = > {
  const userId = parseInt(req.params.id);

  // Найти индекс пользователя
  const userIndex = users.findIndex(u = > u.id === userId);

  if (userIndex === -1) {
    return res.status(404).json({
      error: 'Пользователь не найден'
    });
  }

  // Удаляем пользователя из массива
  const deletedUser = users.splice(userIndex, 1)[0];

  res.status(204).send(); // 204 No Content
  // или если нужно вернуть данные удаленного пользователя:
  // res.json({
  //   message: 'Пользователь удален',
  //   user: deletedUser
  // });
});

app.listen(3000);

Как отправить DELETE запрос:

Через fetch:
fetch('http://localhost:3000/users/1', {
  method: 'DELETE',
  headers: {
    'Content-Type': 'application/json'
  }
})
.then(res = > {
  if (res.status === 204) {
    console.log('Пользователь удален');
  } else {
    return res.json();
  }
})
.then(data = > {
  if (data) console.log(data);
});
Через curl:
curl -X DELETE http://localhost:3000/users/1

Ответ на успешный DELETE запрос:

// Статус 204 (No Content) — нет содержимого ответа
// или

{
  "message": "Пользователь удален",
  "user": {
    "id": 1,
    "name": "Иван",
    "email": "ivan@mail.com"
  }
}

Статус 204 (No Content) — правильный ответ на успешное удаление. Это сообщает клиенту, что операция завершена успешно и нет данных для отправки. Статус 200 с JSON также допустим, если нужно вернуть информацию об удаленном ресурсе.

PATCH vs PUT: Различия

PATCH и PUT — это разные методы обновления. PUT заменяет весь ресурс, PATCH обновляет только переданные поля. Таблица ниже показывает ключевые различия.

Аспект PUT PATCH
Назначение Полное обновление ресурса Частичное обновление ресурса
Обновление полей Все поля заменяются на новые Только переданные поля обновляются
Обязательность данных Требуются все поля объекта Требуются только изменяемые поля
Идемпотентность Идемпотентный (результат не меняется) Может быть неидемпотентным
Пример URL PUT /users/1 PATCH /users/1
Тело запроса { name, email, age, phone } { age } (только изменяемые поля)
Статус ответа 200 (OK) или 204 (No Content) 200 (OK)

Сравнение примеров:

Пример: обновление только возраста пользователя
// ❌ PUT требует все поля
fetch('http://localhost:3000/users/1', {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    name: 'Иван',      // Обязательно передать ВСЕ поля
    email: 'ivan@mail.com',
    age: 26,
    phone: '+7-999-123-45-67'
  })
});

// ✅ PATCH требуе�� только изменяемые поля
fetch('http://localhost:3000/users/1', {
  method: 'PATCH',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    age: 26  // Передаем только поле, которое меняем
  })
});

Когда использовать PATCH:

  • Обновление одного или нескольких полей ресурса.
  • Частичное изменение профиля пользователя (только имя или только email).
  • Изменение состояния ресурса (статус заказа, отметка о прочтении).
  • Экономия трафика — отправляем только нужные данные.

Когда использовать PUT:

  • Полная замена всего ресурса.
  • Обновление всех свойств объекта одновременно.
  • Когда требуется гарантия, что неуказанные поля удаляются или становятся нулевыми.

⚠️ Типичные ошибки:

  • Использование PUT, когда нужен PATCH — отправляем все поля, даже если меняем только одно.
  • Использование PATCH, когда нужен PUT — неправильная семантика операции.
  • Забывание обновить поле updatedAt при PATCH запросе.

Примеры использования PATCH и DELETE

Практические примеры демонстрируют реальные сценарии использования PATCH для обновления отдельных полей и DELETE для удаления ресурсов.

Пример 1: PATCH для изменения статуса заказа

// Хранилище заказов
let orders = [
  { id: 1, productName: 'Ноутбук', status: 'pending', total: 50000, createdAt: '2024-07-14' },
  { id: 2, productName: 'Телефон', status: 'shipped', total: 20000, createdAt: '2024-07-13' }
];

// PATCH для изменения статуса заказа
app.patch('/orders/:id/status', (req, res) = > {
  const orderId = parseInt(req.params.id);
  const { status } = req.body;

  if (!['pending', 'shipped', 'delivered', 'cancelled'].includes(status)) {
    return res.status(400).json({ error: 'Неверный статус' });
  }

  const order = orders.find(o = > o.id === orderId);
  if (!order) return res.status(404).json({ error: 'Заказ не найден' });

  order.status = status;
  order.updatedAt = new Date();

  res.json({
    message: 'Статус заказа обновлен',
    order
  });
});

Пример 2: DELETE с мягким удалением (soft delete)

// Хранилище сообщений с мягким удалением
let messages = [
  { id: 1, text: 'Привет!', author: 'Иван', deletedAt: null },
  { id: 2, text: 'Как дела?', author: 'Мария', deletedAt: null }
];

// DELETE с мягким удалением (не удаляем из БД, а устанавливаем флаг)
app.delete('/messages/:id', (req, res) = > {
  const messageId = parseInt(req.params.id);
  const message = messages.find(m = > m.id === messageId);

  if (!message) return res.status(404).json({ error: 'Сообщение не найдено' });

  if (message.deletedAt) {
    return res.status(410).json({ error: 'Сообщение уже удалено' });
  }

  // Мягкое удаление — устанавливаем флаг удаления
  message.deletedAt = new Date();

  res.json({
    message: 'Сообщение удалено',
    deletedMessage: message
  });
});

// GET для получения только активных сообщений
app.get('/messages', (req, res) = > {
  const activeMessages = messages.filter(m = > m.deletedAt === null);
  res.json(activeMessages);
});

Пример 3: DELETE с проверкой прав доступа

// Хранилище постов
let posts = [
  { id: 1, title: 'Пост 1', author: 'Иван', authorId: 1 },
  { id: 2, title: 'Пост 2', author: 'Мария', authorId: 2 }
];

// Middleware для проверки авторства
const checkAuthor = (req, res, next) = > {
  const postId = parseInt(req.params.id);
  const userId = req.headers['x-user-id'];

  const post = posts.find(p = > p.id === postId);
  if (!post) return res.status(404).json({ error: 'Пост не найден' });

  if (post.authorId !== parseInt(userId)) {
    return res.status(403).json({ error: 'Вы не можете удалить чужой пост' });
  }

  next();
};

// DELETE с проверкой авторства
app.delete('/posts/:id', checkAuthor, (req, res) = > {
  const postId = parseInt(req.params.id);
  const postIndex = posts.findIndex(p = > p.id === postId);

  const deletedPost = posts.splice(postIndex, 1)[0];

  res.json({
    message: 'Пост удален',
    post: deletedPost
  });
});

Совет: Используйте мягкое удаление (soft delete) для важных данных. Это позволяет восстановить удаленные данные и сохранить историю.

Валидация данных при PATCH

Валидация при PATCH отличается от валидации при POST/PUT. При PATCH нужно проверять только переданные поля, а не все поля ресурса.

1. Валидация с проверкой конкретных полей:

// Middleware для валидации PATCH запроса
const validatePatch = (req, res, next) = > {
  const { age, email, phone } = req.body;
  const errors = [];

  if (age !== undefined) {
    if (typeof age !== 'number' || age < 0 || age > 150) {
      errors.push('age должно быть числом от 0 до 150');
    }
  }

  if (email !== undefined) {
    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    if (!emailRegex.test(email)) {
      errors.push('email должно быть валидным');
    }
  }

  if (phone !== undefined) {
    if (typeof phone !== 'string' || phone.length < 10) {
      errors.push('phone должно быть строкой минимум 10 символов');
    }
  }

  if (errors.length > 0) {
    return res.status(400).json({ errors });
  }

  next();
};

app.patch('/users/:id', validatePatch, (req, res) = > {
  // ... обработка
});

2. Валидация с Joi для PATCH:

const Joi = require('joi');

// Схема для PATCH запроса (все поля опциональны)
const patchUserSchema = Joi.object({
  name: Joi.string().min(2).max(50),
  email: Joi.string().email(),
  age: Joi.number().integer().min(0).max(150),
  phone: Joi.string().min(10)
}).min(1); // Минимум одно поле должно быть передано

const validatePatch = (schema) = > {
  return (req, res, next) = > {
    const { error, value } = schema.validate(req.body, { abortEarly: false });

    if (error) {
      return res.status(400).json({
        errors: error.details.map(err = > ({
          field: err.path[0],
          message: err.message
        }))
      });
    }

    req.body = value;
    next();
  };
};

app.patch('/users/:id', validatePatch(patchUserSchema), (req, res) = > {
  // ... обработка
});

Важно при PATCH: Все поля должны быть опциональными, но минимум одно поле должно быть передано. Это предотвращает отправку пустых PATCH запросов.

Обработка ошибок при PATCH и DELETE

Обработка ошибок при PATCH и DELETE требует специальной внимательности, так как эти операции изменяют или удаляют данные.

Типы ошибок:

  • 404 Not Found — ресурс не существует.
  • 400 Bad Request — неверные данные в запросе.
  • 409 Conflict — конфликт (например, попытка обновить удаленный ресурс).
  • 410 Gone — ресурс был удален (для мягкого удаления).
  • 403 Forbidden — нет прав доступа для операции.

1. Обработка PATCH ошибок:

app.patch('/users/:id', (req, res) = > {
  try {
    const userId = parseInt(req.params.id);
    const user = users.find(u = > u.id === userId);

    // Проверка существования
    if (!user) {
      return res.status(404).json({
        error: 'Пользователь не найден'
      });
    }

    // Проверка пустого запроса
    if (Object.keys(req.body).length === 0) {
      return res.status(400).json({
        error: 'Передайте хотя бы одно поле для обновления'
      });
    }

    // Обновление только переданных полей
    Object.keys(req.body).forEach(key = > {
      if (['name', 'email', 'age'].includes(key)) {
        user[key] = req.body[key];
      }
    });

    res.json({
      message: 'Пользователь обновлен',
      user
    });
  } catch (err) {
    console.error('Ошибка при обновлении пользователя:', err);
    res.status(500).json({
      error: 'Внутренняя ошибка сервера'
    });
  }
});

2. Обработка DELETE ошибок:

app.delete('/users/:id', (req, res) = > {
  try {
    const userId = parseInt(req.params.id);
    const userIndex = users.findIndex(u = > u.id === userId);

    // Проверка существования
    if (userIndex === -1) {
      return res.status(404).json({
        error: 'Пользователь не найден'
      });
    }

    // Проверка уже удаленного ресурса (мягкое удаление)
    if (users[userIndex].deletedAt) {
      return res.status(410).json({
        error: 'Пользователь уже был удален'
      });
    }

    // Выполняем удаление
    const deletedUser = users.splice(userIndex, 1)[0];

    res.json({
      message: 'Пользователь удален',
      user: deletedUser
    });
  } catch (err) {
    console.error('Ошибка при удалении пользователя:', err);
    res.status(500).json({
      error: 'Внутренняя ошибка сервера'
    });
  }
});

3. Таблица HTTP статусов для ошибок:

Статус Значение Когда использовать
400 Bad Request Неверные данные в PATCH запросе
403 Forbidden Нет прав удалить/обновить ресурс
404 Not Found Ресурс не существует
409 Conflict Конфликт при обновлении
410 Gone Ресурс был удален (мягкое удаление)
500 Server Error Ошибка на сервере

⚠️ Типичные ошибки обработки:

  • Не проверить существование ресурса перед PATCH/DELETE.
  • Не обрабатывать конфликты при одновременном обновлении.
  • Не проверить права доступа перед удалением.
  • Отправлять 200 вместо 404 при удалении несуществующего ресурса.

Практические примеры

Практические примеры показывают полный CRUD API с поддержкой PATCH и DELETE операций для реальных сценариев.

Пример 1: Полный API для управления товарами с PATCH

const express = require('express');
const app = express();

app.use(express.json());

let products = [
  { id: 1, name: 'Ноутбук', price: 50000, stock: 5, category: 'Электроника' },
  { id: 2, name: 'Телефон', price: 20000, stock: 10, category: 'Электроника' }
];

// PATCH для обновления цены или количества
app.patch('/products/:id', (req, res) = > {
  const product = products.find(p = > p.id === parseInt(req.params.id));
  if (!product) return res.status(404).json({ error: 'Товар не найден' });

  if (Object.keys(req.body).length === 0) {
    return res.status(400).json({ error: 'Передайте хотя бы одно поле' });
  }

  if (req.body.name !== undefined) product.name = req.body.name;
  if (req.body.price !== undefined) product.price = req.body.price;
  if (req.body.stock !== undefined) product.stock = req.body.stock;
  if (req.body.category !== undefined) product.category = req.body.category;

  product.updatedAt = new Date();

  res.json({
    message: 'Товар обновлен',
    product
  });
});

// DELETE для удаления товара
app.delete('/products/:id', (req, res) = > {
  const index = products.findIndex(p = > p.id === parseInt(req.params.id));
  if (index === -1) return res.status(404).json({ error: 'Товар не найден' });

  const deleted = products.splice(index, 1)[0];
  res.json({
    message: 'Товар удален',
    product: deleted
  });
});

app.listen(3000);

Пример 2: API для комментариев с мягким удалением

let comments = [
  { id: 1, text: 'Отличная статья!', author: 'Иван', rating: 5, deletedAt: null },
  { id: 2, text: 'Спасибо за информацию', author: 'Мария', rating: 4, deletedAt: null }

  if (!comment) return res.status(404).json({ error: 'Комментарий не найден' });

  if (comment.deletedAt) {
    return res.status(410).json({ error: 'Комментарий был удален' });
  }

  if (req.body.rating !== undefined) {
    if (req.body.rating < 1 || req.body.rating > 5) {
      return res.status(400).json({ error: 'Рейтинг должен быть от 1 до 5' });
    }
    comment.rating = req.body.rating;
  }

  if (req.body.text !== undefined) {
    comment.text = req.body.text;
  }

  res.json({
    message: 'Комментарий обновлен',
    comment
  });
});

// DELETE с мягким удалением
app.delete('/comments/:id', (req, res) = > {
  const comment = comments.find(c = > c.id === parseInt(req.params.id));
  if (!comment) return res.status(404).json({ error: 'Комментарий не найден' });

  if (comment.deletedAt) {
    return res.status(410).json({ error: 'Комментарий уже удален' });
  }

  comment.deletedAt = new Date();

  res.json({
    message: 'Комментарий удален',
    comment
  });
});

// GET только активные комментарии
app.get('/comments', (req, res) = > {

Безопасность при PATCH и DELETE

Безопасность при PATCH и DELETE критична, так как эти операции изменяют или удаляют данные. Нужна правильная авторизация, валидация и аудит.

Основные угрозы:

  • Несанкционированное удаление — удаление чужих ресурсов.
  • Случайное удаление — удаление важных данных без подтверждения.
  • Mass Assignment — обновление недопустимых полей через PATCH.
  • Race Conditions — одновременное обновление одного ресурса.

1. Проверка прав доступа:

// Middleware для проверки авторства
const checkOwnership = (req, res, next) = > {
  const resourceId = parseInt(req.params.id);
  const userId = req.user.id; // Получаем ID пользователя из токена

  const resource = resources.find(r = > r.id === resourceId);
  if (!resource) return res.status(404).json({ error: 'Ресурс не найден' });

  if (resource.userId !== userId) {
    return res.status(403).json({ error: 'Вы не можете модифицировать чужой ресурс' });
  }

  next();
};

// DELETE с проверкой авторства
app.delete('/resources/:id', checkOwnership, (req, res) = > {
  const index = resources.findIndex(r = > r.id === parseInt(req.params.id));
  const deleted = resources.splice(index, 1)[0];
  res.json({ message: 'Ресурс удален', resource: deleted });
});

2. Whitelist для PATCH (избегаем Mass Assignment):

// Разрешенные поля для обновления
const PATCHABLE_FIELDS = ['name', 'email', 'phone', 'bio'];

// PATCH с проверкой разрешенных полей
app.patch('/users/:id', (req, res) = > {
  const user = users.find(u = > u.id === parseInt(req.params.id));
  if (!user) return res.status(404).json({ error: 'Пользователь не найден' });

  // Обновляем только разрешенные поля
  PATCHABLE_FIELDS.forEach(field = > {
    if (req.body[field] !== undefined) {
      user[field] = req.body[field];
    }
  });

  // Блокируем обновление защищенных полей
  if (req.body.role) {
    return res.status(400).json({ error: 'Нельзя менять роль' });
  }

  res.json({
    message: 'Пользователь обновлен',
    user
  });
});

3. Подтверждение перед критичным удалением:

// DELETE с требованием подтверждения
app.delete('/users/:id', (req, res) = > {
  const confirmToken = req.headers['x-confirm-token'];

  if (!confirmToken) {
    return res.status(400).json({
      error: 'Требуется подтверждение',
      message: 'Передайте x-confirm-token в заголовках'
    });
  }

  // Проверяем токен подтверждения
  if (confirmToken !== 'CONFIRM_DELETE_' + req.params.id) {
    return res.status(403).json({ error: 'Неверный токен подтверждения' });
  }

  const index = users.findIndex(u = > u.id === parseInt(req.params.id));
  if (index === -1) return res.status(404).json({ error: 'Пользователь не найден' });

  const deleted = users.splice(index, 1)[0];
  res.json({ message: 'Пользователь удален', user: deleted });
});

4. Логирование для аудита:

// Middleware для логирования PATCH и DELETE
const auditLog = (req, res, next) = > {
  const originalSend = res.send;

  res.send = function(data) {
    if ((req.method === 'PATCH' || req.method === 'DELETE') && res.statusCode < 400) {
      console.log({
        timestamp: new Date(),
        method: req.method,
        url: req.url,
        userId: req.user?.id,
        status: res.statusCode
      });
    }
    return originalSend.call(this, data);
  };

  next();
};

app.use(auditLog);

⚠️ Правила безопасности:

  • ВСЕГДА проверяйте права доступа перед PATCH и DELETE.
  • Используйте whitelist для разрешенных полей в PATCH.
  • Логируйте все критичные операции удаления.
  • Используйте мягкое удаление для важных данных.
  • Требуйте подтверждение для критичного удаления.

Итоги

  • PATCH обновляет только переданные поля, экономит трафик и более гибкий.
  • DELETE удаляет ресурс по ID, обычно возвращает 204 (No Content).
  • PATCH vs PUT: PATCH для частичного, PUT для полного обновления.
  • Валидация при PATCH: проверяем только переданные поля, минимум одно поле обязательно.
  • Ошибки DELETE: 404 если не найден, 410 если уже удален, 403 если нет прав.
  • Авторизация: проверяйте права доступа перед PATCH и DELETE.
  • Whitelist: ограничивайте разрешенные поля в PATCH (избегаем Mass Assignment).
  • Мягкое удаление: используйте флаг deletedAt вместо физического удаления из БД.
  • Логирование: записывайте все критичные операции для аудита.
