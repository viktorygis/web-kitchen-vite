POST и PUT запросы в Express.js

POST и PUT запросы — это методы для отправки данных на сервер. POST используется для создания новых ресурсов, PUT — для обновления существующих.
Обе используют тело запроса (body) для передачи данных, но имеют разные семантики и применение.

Протоколы передачи данных

  • HTTP — (Hypertext Transfer Protocol - гипертекстовый протокол передачи данных) — стандартный протокол передачи данных в вебе. POST и PUT являются его методами.
  • HTTPS — (Hypertext Transfer Protocol Secure — защищенный гипертекстовый протокол передачи данных) — защищенная версия HTTP, шифрует данные между клиентом и сервером. Всегда используйте HTTPS для передачи чувствительных данных.
  • HTTP/2 — улучшенная версия протокола, поддерживающая мультиплексирование и сжатие заголовков.
  • HTTP/3 — новейшая версия протокола, использующая QUIC для более быстрой и надежной передачи данных.

В этом разделе вы научитесь:

  • Создавать обработчики POST запросов для создания данных.
  • Использовать PUT запросы для полного обновления ресурсов.
  • Работать с телом запроса (req.body).
  • Валидировать и обрабатывать ошибки при работе с данными.
  • Реализовать полноценный REST API.

POST запросы

POST запрос — это HTTP метод для создания нового ресурса на сервере. Данные отправляются в теле запроса, а не в URL. POST используется, когда нужно создать новый объект (пользователя, товар, сообщение и т.д.).

Характеристики POST:

  • Не идемпотентный — каждый POST запрос создает новый ресурс. Два одинаковых POST запроса создадут два разных объекта.
  • Данные в теле — информация передается в теле (body) запроса, не видна в URL.
  • Статус ответа — обычно возвращает 201 (Created) при успехе.
  • Может иметь побочные эффекты — изменяет состояние сервера.

Базовый пример POST запроса:

POST-запросы — это тип HTTP-запроса, который используется для отправки данных на сервер. В отличие от GET-запросов, которые только получают данные, POST-запросы позволяют серверу получать и сохранять данные. 

app.post('/add-user', (req, res) => {
  const user = req.body; // Получаем данные пользователя из тела запроса
  res.send(`Пользователь ${user.name} был добавлен!`);
});
  • app.post — это метод Express для обработки POST-запросов по указанному маршруту.
  • req.body — объект, содержащий данные, отправленные в теле запроса. Для его использования необходимо подключить middleware express.json() или express.urlencoded().
  • res.send — отправляет ответ клиенту. В данном случае мы отправляем сообщение о том, что пользователь был добавлен.

Базовый пример POST запроса:

const express = require('express');
const app = express();

// Middleware для парсинга JSON в теле запроса
app.use(express.json());

// Хранилище пользователей (в реальном приложении будет БД)
const users = [];

// POST обработчик — создание нового пользователя
app.post('/users', (req, res) => {
  const { name, email, age } = req.body;

  // Создаем новый объект
  const newUser = {
    id: users.length + 1,
    name,
    email,
    age,
    createdAt: new Date()
  };

  users.push(newUser);

  // Возвращаем созданный ресурс со статусом 201
  res.status(201).json({
    message: 'Пользователь успешно создан',
    user: newUser
  });
});

app.listen(3000);

Как отправить POST запрос:

Способ 1: Через fetch в браузере:
fetch('http://localhost:3000/users', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    name: 'Иван',
    email: 'ivan@example.com',
    age: 25
  })
})
.then(res => res.json())
.then(data => console.log(data));
Способ 2: Через curl в терминале:
curl -X POST http://localhost:3000/users \
  -H "Content-Type: application/json" \
  -d '{"name":"Иван","email":"ivan@example.com","age":25}'
Способ 3: Через fetch с форм-данными:
// Отправить данные формы
const formData = new FormData(document.querySelector('form'));

fetch('http://localhost:3000/users', {
  method: 'POST',
  body: formData
})
.then(res => res.json())
.then(data => console.log(data));

Ответ сервера на успешный POST запрос:

{
  "message": "Пользователь успешно создан",
  "user": {
    "id": 1,
    "name": "Иван",
    "email": "ivan@example.com",
    "age": 25,
    "createdAt": "2024-07-14T10:30:00.000Z"
  }
}

Статус 201 (Created) — правильный ответ на успешное создание ресурса. Браузер и клиенты знают, что ресурс был создан, а не просто обновлен.

PUT запросы

PUT запрос — это HTTP метод для полного обновления существующего ресурса. Данные отправляются в теле запроса, как и в POST, но PUT предполагает замену всего объекта. ID ресурса обычно передается в URL пути.

Характеристики PUT:

  • Идемпотентный — несколько одинаковых PUT запросов приводят к одному и тому же результату.
  • Полное обновление — заменяет весь ресурс на новые данные.
  • Требует ID — ID ресурса передается в URL (обычно /resource/:id).
  • Статус ответа — обычно возвращает 200 (OK) или 204 (No Content).
  • Может создать ресурс — если ресурс не существует, может его создать (но обычно возвращает 404).

Базовый пример PUT запроса:

В отличие от POST-запросов, которые добавляют новые данные, ❕ PUT-запросы заменяют старые данные новыми. Обычно PUT-запросы отправляются, когда нужно обновить конкретный ресурс. 

app.put('/update-user/:id', (req, res) => {
  const userId = req.params.id; // Получаем ID пользователя из параметра маршрута
  const updatedData = req.body; // Получаем обновленные данные из тела запроса

  res.send(`Пользователь с ID ${userId} был обновлён! Новые данные: ${JSON.stringify(updatedData)}`);
});
  • app.put — это метод Express для обработки PUT-запросов по указанному маршруту.
  • req.params.id — содержит ID ресурса из URL пути.
  • req.body — объект, содержащий данные, отправленные в теле запроса. Для его использования необходимо подключить middleware express.json() или express.urlencoded().
  • res.send — отправляет ответ клиенту. В данном случае мы отправляем сообщение о том, что пользователь был обновлён.

Частичное обновление ресурса

PUT-запросы обычно используются для полного обновления ресурса, однако мы можем использовать их для частичного обновления, если нужно обновить только часть данных. Например, мы можем обновить только название или цену товара, оставив другие данные без изменений. 

app.put('/update-product/:id', (req, res) => {
  const productId = req.params.id; // Получаем ID продукта из URL
  const updatedData = req.body; // Получаем обновленные данные из тела запроса

  const product = {
    name: updatedData.name || 'Неизвестный продукт', // Если имя не передано, оставляем дефолтное значение
    price: updatedData.price || 0, // Если цена не передана, оставляем дефолтное значение
    description: updatedData.description || 'Нет описания' // Если описание не передано, оставляем дефолтное
  };

  res.send(`Продукт с ID ${productId} был обновлён! Новые данные: ${JSON.stringify(product)}`);
});

Здесь мы обновляем только те данные, которые были переданы в запросе. Если какого-то поля нет в запросе, то оно остаётся с дефолтным значением. 💡 Запомни! В PUT-запросах для частичного обновления можно использовать условные операторы (например, ||), чтобы оставить старые значения, если новые данные не были переданы.

Базовый пример PUT запроса:

// Хранилище пользователей
let users = [
  { id: 1, name: 'Иван', email: 'ivan@mail.com', age: 25 },
  { id: 2, name: 'Мария', email: 'maria@mail.com', age: 30 }
];

// PUT обработчик — полное обновление пользователя
app.put('/users/:id', (req, res) => {
  const userId = parseInt(req.params.id);
  const { name, email, age } = req.body;

  // Найти пользователя по ID
  const userIndex = users.findIndex(u => u.id === userId);

  if (userIndex === -1) {
    return res.status(404).json({
      error: 'Пользователь не найден'
    });
  }

  // Полностью заменяем данные пользователя
  users[userIndex] = {
    id: userId,
    name,
    email,
    age,
    updatedAt: new Date()
  };

  res.json({
    message: 'Пользователь обновлен',
    user: users[userIndex]
  });
});

app.listen(3000);

Как отправить PUT запрос:

Через fetch:
fetch('http://localhost:3000/users/1', {
  method: 'PUT',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    name: 'Иван Иванов',
    email: 'ivan.new@example.com',
    age: 26
  })
})
.then(res => res.json())
.then(data => console.log(data));
Через curl:
curl -X PUT http://localhost:3000/users/1 \
  -H "Content-Type: application/json" \
  -d '{"name":"Иван Иванов","email":"ivan.new@example.com","age":26}'

Ответ на успешный PUT запрос:

{
  "message": "Пользователь обновлен",
  "user": {
    "id": 1,
    "name": "Иван Иванов",
    "email": "ivan.new@example.com",
    "age": 26,
    "updatedAt": "2024-07-14T11:45:00.000Z"
  }
}

Идемпотентность PUT: Отправить одинаковый PUT запрос 10 раз — результат будет одинаковым. При этом POST создаст 10 разных ресурсов.

POST vs PUT: Различия

POST и PUT часто путают, но они решают разные задачи. Таблица ниже показывает ключевые различия.

Аспект POST PUT
Назначение Создать новый ресурс Обновить существующий ресурс
Идемпотентность Не идемпотентный (10 запросов = 10 ресурсов) Идемпотентный (10 запросов = 1 результат)
ID ресурса Генерируется сервером, не передается Передается в URL пути (/resource/:id)
Статус ответа 201 (Created) 200 (OK) или 204 (No Content)
Пример URL POST /users PUT /users/1
Пример действия Создать нового пользователя Обновить пользовател�� с ID 1
Тело запроса Основные данные нового ресурса Полные или частичные данные для обновления

Сравнение примеров:

❌ Неправильное использование POST (вместо PUT):
// ❌ НЕПРАВИЛЬНО — используем POST для обновления
fetch('http://localhost:3000/users/1', {
  method: 'POST', // Должен быть PUT!
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'Новое имя' })
});

// Проблема: каждый запрос создаст новый ресурс вместо обновления
✅ Правильное использование POST и PUT:
// ✅ POST для создания
fetch('http://localhost:3000/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'Иван', email: 'ivan@mail.com' })
});

// ✅ PUT для обновления
fetch('http://localhost:3000/users/1', {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'Иван Обновленный', email: 'ivan.new@mail.com' })
});

Сценарии использования:

Когда использовать POST:
  • Создание новой записи в базе данных.
  • Регистрация нового пользователя.
  • Добавление товара в корзину.
  • Публикация нового поста или комментария.
  • Загрузка файла на сервер.
Когда использовать PUT:
  • Полное обновление профиля пользователя.
  • Изменение всех свойств товара (название, цена, описание).
  • Замена содержимого статьи.
  • Обновление настроек приложения.

⚠️ Типичная ошибка:

  • Использование POST вместо PUT для обновления — каждый запрос создает новый ресурс.
  • Забывание ID в URL при PUT — PUT /users вместо PUT /users/:id.
  • Отправка одного PUT запроса и ожидание создания ресурса — PUT обновляет существующий.

Работа с телом запроса (req.body)

req.body — это объект, содержащий данные из тела HTTP запроса. Для парсинга этих данных нужно подключить соответствующий middleware Express.

Типы данных в теле запроса:

  • JSON — application/json — парсится middleware express.json().
  • Данные формы — application/x-www-form-urlencoded — парсится middleware express.urlencoded().
  • Файлы — multipart/form-data — требует специального middleware (multer).

1. Работа с JSON:

const express = require('express');
const app = express();

// Middleware для парсинга JSON
app.use(express.json());

app.post('/users', (req, res) => {
  // req.body содержит распарсенные JSON данные
  console.log(req.body); // { name: 'Иван', age: 25 }

  const { name, age } = req.body;

  res.status(201).json({
    message: 'Пользователь создан',
    data: { name, age }
  });
});

app.listen(3000);
Отправка JSON через fetch:
fetch('http://localhost:3000/users', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    name: 'Иван',
    age: 25
  })
});

2. Работа с данными формы:

// Middleware для парсинга данных формы
app.use(express.urlencoded({ extended: true }));

app.post('/login', (req, res) => {
  const { username, password } = req.body;

  res.json({
    message: 'Данные получены',
    username,
    password
  });
});
Отправка данных формы:
// HTML форма
<form method="POST" action="/login">
  <input type="text" name="username" required />
  <input type="password" name="password" required />
  <button type="submit">Вход</button>
</form>

3. Работа с файлами (FormData):

// Отправка файла через FormData
const form = document.querySelector('form');
const formData = new FormData(form);

fetch('http://localhost:3000/upload', {
  method: 'POST',
  body: formData // Автоматически устанавливает multipart/form-data
});

4. Ограничение размера тела запроса:

// Ограничить размер JSON до 1MB
app.use(express.json({ limit: '1mb' }));

// Ограничить размер данных формы до 2MB
app.use(express.urlencoded({ limit: '2mb', extended: true }));

// Обработчик ошибки при превышении лимита
app.use((err, req, res, next) => {
  if (err instanceof SyntaxError && err.status === 400 && 'body' in err) {
    return res.status(413).json({ error: 'Тело запроса слишком большое' });
  }
  next();
});

5. Получение сырых данных:

// Получить сырые данные без парсинга
app.post('/raw', express.raw({ type: 'application/octet-stream' }), (req, res) => {
  console.log(req.body); // Buffer с сырыми данными
  res.send('Данные получены');
});

Важно! Middleware для парсинга должны быть подключены ДО маршрутов, которые их используют. Порядок: middleware → маршруты.

Настройка middleware для POST и PUT

Middleware для обработки данных — это функции Express, которые парсят тело запроса и делают данные доступными в req.body. Правильная настройка критична для работы POST и PUT.

Полная настройка middleware:

const express = require('express');
const app = express();

// 1. Middleware для парсинга JSON
app.use(express.json());

// 2. Middleware для парсинга данных формы
app.use(express.urlencoded({ extended: true }));

// 3. Middleware для парсинга текста
app.use(express.text());

// 4. Custom middleware для логирования
app.use((req, res, next) => {
  if (req.method === 'POST' || req.method === 'PUT') {
    console.log(`📨 ${req.method} ${req.url}`);
    console.log('Body:', req.body);
  }
  next();
});

// Теперь все маршруты имеют доступ к req.body
app.post('/users', (req, res) => {
  res.json(req.body);
});

app.listen(3000);

Middleware для конкретного маршрута:

// Применить middleware только для определенного маршрута
app.post('/users', express.json(), (req, res) => {
  res.json(req.body);
});

// Применить несколько middleware
app.put('/users/:id',
  express.json(),
  (req, res, next) => {
    console.log('Custom middleware');
    next();
  },
  (req, res) => {
    res.json(req.body);
  }
);

Custom middleware для парсинга:

// Custom middleware для проверки типа контента
const checkContentType = (req, res, next) => {
  const contentType = req.headers['content-type'];

  if (req.method === 'POST' || req.method === 'PUT') {
    if (!contentType || !contentType.includes('application/json')) {
      return res.status(415).json({
        error: 'Content-Type должен быть application/json'
      });
    }
  }

  next();
};

app.use(checkContentType);
app.use(express.json());

app.post('/users', (req, res) => {
  res.json(req.body);
});

Обработка ошибок парсинга:

app.use(express.json());

// Error middleware для обработки ошибок парсинга
app.use((err, req, res, next) => {
  if (err instanceof SyntaxError && err.status === 400 && 'body' in err) {
    return res.status(400).json({
      error: 'Невалидный JSON в теле запроса',
      details: err.message
    });
  }
  next(err);
});

Порядок middleware: Подключайте middleware для парсинга В САМОМ НАЧАЛЕ приложения, даже перед маршрутами. Это гарантирует, что все маршруты получат распарсенные данные.

Валидация данных

Валидация — это проверка корректности данных, полученных от клиента. Валидировать нужно всегда, даже если вы уверены в клиенте. Валидация защищает приложение от ошибок и атак.

Основные правила валидации:

  • Проверяйте, что все обязательные поля присутствуют.
  • Проверяйте типы данных (строка, число, булево и т.д.).
  • Проверяйте формат данных (email, URL, телефон и т.д.).
  • Проверяйте длину строк и диапазон чисел.
  • Санитизируйте данные — удаляйте опасный контент.

1. Простая валидация:

app.post('/users', (req, res) => {
  const { name, email, age } = req.body;

  // Проверка обязательных полей
  if (!name || !email || age === undefined) {
    return res.status(400).json({
      error: 'Требуются поля: name, email, age'
    });
  }

  // Проверка типов
  if (typeof name !== 'string' || typeof email !== 'string' || typeof age !== 'number') {
    return res.status(400).json({
      error: 'Неверные типы данных: name(string), email(string), age(number)'
    });
  }

  // Проверка диапазонов
  if (age < 0 || age > 150) {
    return res.status(400).json({
      error: 'Возраст должен быть от 0 до 150'
    });
  }

  // Проверка формата email
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  if (!emailRegex.test(email)) {
    return res.status(400).json({
      error: 'Некорректный email'
    });
  }

  res.status(201).json({ name, email, age });
});

2. Middleware для валидации:

// Middleware для валидации пользователя
const validateUser = (req, res, next) => {
  const { name, email, age } = req.body;
  const errors = [];

  if (!name || typeof name !== 'string') {
    errors.push('name обязательно и должно быть строкой');
  }

  if (!email || !email.includes('@')) {
    errors.push('email обязательно и должно быть валидным');
  }

  if (age === undefined || typeof age !== 'number' || age < 0) {
    errors.push('age обязательно, должно быть числом и >= 0');
  }

  if (errors.length > 0) {
    return res.status(400).json({ errors });
  }

  next();
};

// Использование middleware

    next();
  };
};

Обработка ошибок

Обработка ошибок при POST и PUT критична для создания надежного API. Нужно обрабатывать ошибки парсинга, валидации, логики приложения и базы данных.

Типы ошибок:

  • Ошибки синтаксиса — неправильный JSON (400 Bad Request).
  • Ошибки валидации — неверные данные (400 Bad Request).
  • Ошибки авторизации — нет прав доступа (401/403).
  • Ошибки ресурса — ресурс не найден (404 Not Found).
  • Ошибки сервера — ошибка в коде или БД (500 Server Error).

1. Базовая обработка ошибок:

app.post('/users', (req, res) => {
  try {
    const { name, email } = req.body;

    // Валидация
    if (!name || !email) {
      return res.status(400).json({
        error: 'Name и email обязательны'
      });
    }

    // Создание пользователя (может быть ошибка БД)
    const user = createUser(name, email);

    res.status(201).json(user);
  } catch (err) {
    // Обработка неожиданных ошибок
    console.error('Ошибка при создании пользователя:', err);
    res.status(500).json({
      error: 'Внутренняя ошибка сервера'
    });
  }
});

2. Обработка ошибок парсинга JSON:

app.use(express.json());

// Middleware для обработки ошибок парсинга
app.use((err, req, res, next) => {
  if (err instanceof SyntaxError && err.status === 400 && 'body' in err) {
    return res.status(400).json({
      error: 'Невалидный JSON',
      details: err.message
    });
  }
  next(err);
});

3. Центральный error middleware:

// Определить класс для API ошибок
class ApiError extends Error {
  constructor(status, message, details = null) {
    super(message);
    this.status = status;
    this.details = details;
  }
}

// Маршруты с выбросом ошибок
app.post('/users', (req, res, next) => {
  try {
    const { name, email } = req.body;

    if (!name) {
      throw new ApiError(400, 'Name обязателен');
    }

    // ... остальной код
    res.status(201).json({ name, email });
  } catch (err) {
    next(err); // Передать ошибку в error middleware
  }
});

// Центральный error middleware
app.use((err, req, res, next) => {
  const status = err.status || 500;
  const message = err.message || 'Внутренняя ошибка сервера';

  console.error(`❌ [${status}] ${message}`, err);

  res.status(status).json({
    error: message,
    status,
    details: err.details || undefined
  });
});

4. Обработка асинхронных ошибок:

// Wrapper для async маршрутов
const asyncHandler = (fn) => (req, res, next) => {
  Promise.resolve(fn(req, res, next)).catch(next);
};

// Использование
app.post('/users', asyncHandler(async (req, res) => {
  const user = await User.create(req.body);
  res.status(201).json(user);
}));

app.put('/users/:id', asyncHandler(async (req, res) => {
  const user = await User.findByIdAndUpdate(req.params.id, req.body);
  if (!user) throw new ApiError(404, 'User not found');
  res.json(user);
}));

5. Правильные HTTP статусы ошибок:

Статус Ошибка Когда использовать
400 Bad Request Невалидный запрос, неверные данные
401 Unauthorized Требуется авторизация
403 Forbidden Нет прав доступа
404 Not Found Ресурс не найден
409 Conflict Конфликт (например, email уже существует)
422 Unprocessable Entity Невозможно обработать (ошибка валидации)
500 Server Error Ошибка на сервере

⚠️ Типичные ошибки обработки ошибок:

  • Не выбрасывать ошибку или не передавать в next() — запрос зависнет.
  • Отправлять ответ дважды — "Cannot set headers after they are sent".
  • Не логировать ошибки — потом невозможно найти проблему.
  • Отправлять сложные детали ошибки клиенту — информация об утечке.

Практические примеры

Практические примеры показывают, как создать простой но полнофункциональный REST API для управления ресурсами (товары, пользователи, посты и т.д.).

Пример 1: Полный CRUD API для товаров

const express = require('express');
const app = express();

app.use(express.json());

// Хранилище товаров
let products = [
  { id: 1, name: 'Ноутбук', price: 50000, stock: 5 },
  { id: 2, name: 'Телефон', price: 20000, stock: 10 }
];

// GET - получить все товары
app.get('/products', (req, res) => {
  res.json(products);
});

// GET - получить товар по ID
app.get('/products/:id', (req, res) => {
  const product = products.find(p => p.id === parseInt(req.params.id));
  if (!product) return res.status(404).json({ error: 'Товар не найден' });
  res.json(product);
});

// POST - создать новый товар
app.post('/products', (req, res) => {
  const { name, price, stock } = req.body;

  if (!name || price === undefined || stock === undefined) {
    return res.status(400).json({ error: 'Требуются: name, price, stock' });
  }

  const newProduct = {
    id: Math.max(...products.map(p => p.id), 0) + 1,
    name,
    price,
    stock
  };

  products.push(newProduct);
  res.status(201).json(newProduct);
});

// PUT - обновить товар
app.put('/products/:id', (req, res) => {
  const product = products.find(p => p.id === parseInt(req.params.id));
  if (!product) return res.status(404).json({ error: 'Товар не найден' });

  const { name, price, stock } = req.body;

  product.name = name || product.name;
  product.price = price !== undefined ? price : product.price;
  product.stock = stock !== undefined ? stock : product.stock;

  res.json(product);
});

// DELETE - удалить товар
app.delete('/products/:id', (req, res) => {
  const index = products.findIndex(p => p.id === parseInt(req.params.id));
  if (index === -1) return res.status(404).json({ error: 'Товар не найден' });

  const deleted = products.splice(index, 1)[0];
  res.json({ message: 'Товар удален', product: deleted });
});

app.listen(3000, () => console.log('API запущен на порту 3000'));

// Middleware для валидации создания поста

Безопасность при работе с POST и PUT

Безопасность критична при обработке POST и PUT запросов, так как они изменяют данные сервера. Нужно защищаться от различных атак и уязвимостей.

Основные угрозы безопасности:

  • SQL Injection — внедрение SQL кода в данные.
  • XSS (Cross-Site Scripting) — внедрение JavaScript кода.
  • CSRF (Cross-Site Request Forgery) — несанкционированные действия от пользователя.
  • DoS (Denial of Service) — перегрузка сервера большими запросами.

1. Валидация и санитизация входных данных:

// npm install sanitize-html xss
const sanitizeHtml = require('sanitize-html');

app.post('/posts', (req, res) => {
  let { title, content } = req.body;

  // Санитизировать опасный контент
  content = sanitizeHtml(content, {
    allowedTags: ['b', 'i', 'em', 'strong', 'a'],
    allowedAttributes: { 'a': ['href'] }
  });

  // Избежать SQL injection с использованием параметризованных запросов
  // db.query('INSERT INTO posts (title, content) VALUES (?, ?)', [title, content]);

  res.status(201).json({ title, content });
});

2. Ограничение размера тела запроса:

// Защита от больших payload'ов
app.use(express.json({ limit: '1mb' }));
app.use(express.urlencoded({ limit: '1mb', extended: true }));

// Custom middleware для проверки размера
const checkBodySize = (req, res, next) => {
  const contentLength = parseInt(req.headers['content-length']) || 0;
  if (contentLength > 1024 * 1024) { // 1MB
    return res.status(413).json({ error: 'Тело запроса слишком большое' });
  }
  next();
};

app.use(checkBodySize);

3. Защита от CSRF атак:

// npm install csurf
const csrf = require('csurf');
const cookieParser = require('cookie-parser');

      <button type="submit">Отправить</button>

// POST с проверкой CSRF токена

  res.json({ message: 'CSRF токен валиден', data: req.body });

5. Аутентификация и авторизация:

// Middleware для проверки аутентификации
const authenticate = (req, res, next) => {
  const token = req.headers['authorization'];

  if (!token) {
    return res.status(401).json({ error: 'Требуется токен авторизации' });

    return res.status(403).json({ error: 'Неверный токен' });

  next();
};

// Использование в маршрутах

  res.status(201).json(req.body);
});

  res.json(req.body);

⚠️ Правила безопасности:

  • ВСЕГДА валидируйте входные данные на сервере, не доверяйте клиенту.
  • Санитизируйте данные перед сохранением в БД.
  • Используйте параметризованные запросы к БД (избегайте конкатенации строк).
  • Ограничивайте частоту запросов (rate limiting).
  • Используйте HTTPS для всех запросов с данными.
  • Логируйте подозрительную активность.

Итоги

  • POST создает новые ресурсы, не идемпотентный, возвращает 201 (Created).
  • PUT обновляет существующие ресурсы полностью, идемпотентный, возвращает 200 (OK).
  • req.body содержит данные из тела запроса, требует подходящего middleware.
  • Middleware для парсинга должны быть подключены ДО маршрутов.
  • Валидация критична — проверяйте все входные данные на сервере.
  • Обработка ошибок — используйте try-catch и error middleware для надежности.
  • Безопасность — санитизируйте данные, ограничивайте размер, используйте аутентификацию.
  • Правильные статусы — 201 для создания, 200 для обновления, 400 для ошибок клиента, 500 для сервера.
  • REST API следует стандартам: используйте правильные методы и статусы.
