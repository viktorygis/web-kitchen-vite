Строгий режим — "use strict"

Строгий режим в JavaScript был введен с появлением ECMAScript 5 в 2009 году для устранения проблем с обратной совместимостью.
Он помогает избежать ошибок и улучшить безопасность кода.

Основные цели:

  • Устранение "небрежных" ошибок
  • Предотвращение использования устаревших возможностей
  • Упрощение оптимизации кода

Где применяется:

  • В отдельных скриптах
  • В функциях
  • Автоматически в модулях и классах

Использование "use strict"

Директива "use strict" — активирует строгий режим, который меняет поведение некоторых функций.
Должна быть первой исполняемой строкой в скрипте или функции.

Глобальное применение:

"use strict";
// Весь код ниже будет в строгом режиме
let x = 10;

Локальное применение:

function strictFunction() {
  'use strict';
  // Только эта функция в строгом режиме
}

Особенности активации:

Ситуация Поведение
Не первая строка Строгий режим не активируется
В консоли браузера Требуется обернуть в IIFE
В модулях/классах Активирован по умолчанию

Особенности строгого режима

Запрещенные действия

Строгий режим запрещает потенциально проблемные конструкции: 

// Без strict:
x = 10; // Создает глобальную переменную

// Со strict:
"use strict";
x = 10; // ReferenceError: x is not defined

Изменения в поведении

// Без strict:
function fn(a, a) { // Дублирование параметров
  console.log(a);
}

// Со strict:
"use strict";
function fn(a, a) { // SyntaxError
  // ...
}

Другие изменения:

  • Запрет на удаление обычных переменных
  • Ограничения на использование eval
  • Запрет на использование with
  • Изменения в работе this

Рекомендации по использованию

Современная практика — всегда использовать строгий режим для нового кода.
В будущих версиях JavaScript многие его ограничения станут стандартом.

Когда использовать:

// Для всего скрипта
"use strict";
// ... весь код

// Или для отдельных функций
function strictFn() {
  'use strict';
  // ...
}

Консоль разработчика:

(function() {
  'use strict';
  // Ваш код для консоли
})();

Лучшие практики:

  1. Всегда включать в начале новых скриптов
  2. Для старых проектов - постепенный переход
  3. В модулях можно не указывать (включен по умолчанию)
  4. Тестировать код в строгом режиме
